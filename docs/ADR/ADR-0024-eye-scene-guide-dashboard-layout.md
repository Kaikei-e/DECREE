# ADR-0024: eye プロジェクト詳細ページ Scene Guide & ダッシュボードレイアウト

## ステータス

Accepted

## 日付

2026-03-16

## コンテキスト

eye サービスのプロジェクト詳細ページ（`/projects/[projectId]`）は、3D/2D 可視化キャンバスの上に HUD オーバーレイとして UI 要素を重ねるレイアウトを採用していた。このアプローチには以下の課題があった:

- **可読性**: オーバーレイ要素がキャンバス上に浮遊するため、背景のノード色と重なった際にテキストが読みにくくなる
- **情報構造の欠如**: フィルターバー・リスク一覧・凡例がキャンバスの四隅に散在し、視線の流れが定まらない
- **可視化の解釈負荷**: 初見のユーザーが「orb の高さ」「色」「明るさ」の意味を理解するための導線がなく、暗黙知に依存していた
- **シーンレベルの要約がない**: 表示中のフィンディング数・重大度分布・最大クラスタなどの俯瞰情報が存在せず、ユーザーは個別ノードを順に確認する必要があった

これらの課題を解決するため、レイアウトの構造化とシーン要約機能の新設を行う必要があった。

## 決定事項

HUD オーバーレイ方式を廃止し、CSS Grid ベースの構造化ダッシュボードレイアウトへ移行する。新規コンポーネント `SceneGuide` を導入し、ビジュアルエンコーディングの凡例とシーンレベルの統計サマリーを常時表示する。統計算出ロジックは `insights.ts` として可視化レイヤーから分離する。

## 実装の概要

### 新規ファイル

**`services/eye/src/lib/graph/insights.ts`**

`buildVisualizationInsights()` 関数が `Finding[]` と `GraphModel` を受け取り、以下を算出する:
- 総フィンディング数・アクティブ数・ターゲット数
- CRITICAL 件数・最高 DECREE Score
- パルス中ノード数（直近 24 時間の観測）
- 最大クラスタ（名前・ノード数）
- 重大度別の内訳（色情報付き）

`getTopVisibleRisks()` 関数は DECREE Score 降順でフィンディングをソートし、上位 N 件を返す。

**`services/eye/src/lib/graph/insights.test.ts`**

TDD で作成。`computeLayout()` で実際の `GraphModel` を生成し、`buildVisualizationInsights()` の各フィールドを検証する。`getTopVisibleRisks()` のソート・リミットも検証。

**`services/eye/src/lib/components/SceneGuide.svelte`**

2 カラムのグリッドレイアウト:
- 左カラム: Scene Map セクション — フィンディング数・ターゲット数・CRITICAL 件数・パルス件数を KPI カードで表示
- 右カラム: Visual Encoding セクション — orb・高さ・色＋明るさの意味を凡例として表示、重大度別のバーチャートを表示

### 変更ファイル

**`services/eye/src/lib/components/FilterBar.svelte`**

- 各フィルターフィールドに `<label>` とヘッダーテキストを追加
- 「Viewport Controls」の説明テキストを追加
- リセットボタン（`RotateCcw` アイコン付き）を追加
- レンダラー切替ボタンに `Box` / `Monitor` アイコンとテキストラベルを付与

**`services/eye/src/lib/components/TopRisksSummary.svelte`**

- ランキング番号（`#1`, `#2`, ...）を各行に表示
- パッケージ名・バージョン・ターゲット名・エコシステムの詳細行を追加
- 件数バッジ（`N shown`）をヘッダー右に表示
- フィルター結果が空の場合の空状態メッセージを追加

**`services/eye/src/routes/projects/[projectId]/+page.svelte`**

- オーバーレイ配置を廃止し、`flex-col` + `grid` による構造化レイアウトへ移行
- `SceneGuide` をキャンバス上部に配置
- `insights.ts` の関数を `$derived` で呼び出し、リアクティブにサマリーを更新
- キャンバス内にインライン凡例チップ（Clusters / Height / Brightness）を残置

**`services/eye/src/app.css`**

- `:root` にダークモード背景グラデーションを追加
- `body` に `min-height: 100vh` と透過背景を設定
- `input[type="range"]` のスライダーに丸みのあるスタイル（`border-radius: 999px`）を適用
- `.hud-panel` に `box-shadow` の深度を追加

### データモデルへの影響

なし。フロントエンド（eye）のみの変更であり、fact / resource / projection いずれの層にも影響しない。

### 可視化への影響

Three.js WebGPURenderer / Sigma.js フォールバック双方のレンダリングロジックには変更なし。変更はレイアウトと UI コンポーネント層に限定される。`insights.ts` は `GraphModel` を読み取り専用で参照するのみ。

## 代替案

### A: オーバーレイの改善（背景ぼかし強化）

オーバーレイ要素の `backdrop-blur` を強化し、背景との干渉を軽減する案。レイアウト変更が不要で影響範囲が小さい。しかし根本的な情報構造の問題（視線の流れが定まらない、シーン要約がない）は解決できないため不採用。

### B: サイドバー固定レイアウト

キャンバスの左右にサイドバーを固定配置する案。デスクトップでは有効だが、狭い画面で可視化領域が圧迫される。Grid ベースであればレスポンシブに折り畳めるため、Grid レイアウトを選択した。

### C: insights ロジックをコンポーネント内に直接記述

`SceneGuide.svelte` 内で直接算出する案。コンポーネントが肥大化し、テスト困難になるため、`insights.ts` として分離した。TDD ワークフローとの親和性も高い。

## 影響・トレードオフ

### メリット

- ユーザーがシーンの構造を即座に把握できる（KPI カード + 凡例）
- フィルター操作の発見性が向上（ラベル付きフィールド + リセットボタン）
- 可視化の読み方が自明になり、オンボーディングコストが低下
- `insights.ts` が純粋関数として分離されているため、単体テストが容易

### デメリット・リスク

- キャンバスの表示領域が SceneGuide の高さ分だけ縮小する（特にモバイル）
- コンポーネント数が増加（SceneGuide 追加）し、バンドルサイズが微増する

### パフォーマンスへの影響

`buildVisualizationInsights()` は `$derived` でリアクティブに呼ばれるが、計算量は O(N)（N = フィンディング数）であり、実用的なデータ量（数千件）では無視できる。DOM 更新もテキスト値のみのため、レンダリング負荷は極めて低い。

## 今後の課題

- SceneGuide の KPI カードにクリックでフィルター適用する機能の追加
- 重大度バーチャートのアニメーション（フィルター変更時のトランジション）
- モバイルビューポートでの SceneGuide 折り畳み・アコーディオン化
- `insights.ts` への時系列トレンド算出の追加（直近 N 日間の変化率）
