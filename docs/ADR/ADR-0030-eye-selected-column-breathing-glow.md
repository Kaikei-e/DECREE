# ADR-0030: 選択カラムのブリージンググローエフェクト

## ステータス

Accepted

## 日付

2026-03-17

## コンテキスト

Threat Skyline（3D可視化）において CVE カラムをクリックすると、DetailPanel や NodeTooltip などの UI サイドバーに選択情報が表示される。しかし、3D シーン上のカラム自体には **視覚的フィードバックが存在しなかった**。

これはカメラをオービット操作して視点を変えた際に「どのカラムを選択中か」を見失う原因となっていた。特にカラムが密集するディストリクト内では、選択状態を把握するために毎回サイドバーに視線を移す必要があり、空間的な文脈が途切れていた。

解決にあたっては、白熱灯のような温かみのある光（約 2700K タングステン色調）でカラムがゆっくり「呼吸」する（パルスする）エフェクトを目指した。

## 決定事項

選択されたカラムの上に、専用のオーバーレイ `Mesh` を配置し、エミッシブマテリアルの発光強度を正弦波で変調させるブリージンググローエフェクトを実装する。

## 実装の概要

### アーキテクチャ

InstancedMesh はマテリアルを全インスタンスで共有するため、選択カラムだけにエミッシブ変更を適用できない。そこで、選択時に **非インスタンスの独立した `THREE.Mesh`** を生成し、元カラムの位置・スケールを複製してシーンに追加する方式を採用した。

```
[InstancedMesh: 全カラム] ← 既存（色 = severity、高さ = DECREE Score）
          ↑
[Mesh: glow-overlay]     ← 新規（1.03× スケール、半透明、エミッシブ呼吸）
```

### 関連サービスと変更ファイル

影響は `decree-eye`（SvelteKit + Three.js）のみ。

| ファイル | 変更内容 |
|---|---|
| `node-material.ts` | グロー定数（`GLOW_COLOR`, `GLOW_PERIOD`, `GLOW_MIN/MAX_INTENSITY`）と `createGlowMaterial()` ファクトリを追加 |
| `types.ts` | `SceneRenderer` インターフェースに `setSelectedNode(nodeId)` を追加 |
| `ThreeSceneRenderer.ts` | `glowMesh` / `glowMaterial` フィールド、`setSelectedNode()` メソッド、`updateGlow()` を追加。`animate()` ループ内で毎フレーム発光強度を更新 |
| `Canvas2DRenderer.ts` | `setSelectedNode()` の no-op スタブを追加（2D フォールバックでは未実装） |
| `VisualizationCanvas.svelte` | `selectedNodeId` プロップと `$effect` でレンダラーへの橋渡し |
| `+page.svelte` | `appState.selectedNodeId` をキャンバスコンポーネントに伝播 |

### グローパラメータ

| パラメータ | 値 | 根拠 |
|---|---|---|
| 色 | `#FFAA44`（0xFFAA44） | 約 2700K の温白色。シーンの寒色系パレット（青/シアン）と補色関係にあり、視認性が高い |
| 呼吸周期 | 3.5 秒 | Apple のスリープインジケータパターンに準拠。気づくが煩わしくない速度 |
| 発光強度 | 0.15 〜 0.7 | 最小値でも存在を示し、最大値で severity カラーエンコーディングを損なわない範囲 |
| オーバーレイスケール | 1.03× | Z ファイティングを防止しつつ、元カラムのシルエットを大きく逸脱しない |
| 透明度 | 0.85 | 元カラムの severity 色がわずかに透けて、情報量を保持 |
| `depthWrite` | `false` | 半透明オーバーレイの描画順序問題を回避 |

### 正弦波変調

```
t = (elapsed % GLOW_PERIOD) / GLOW_PERIOD
sine = (sin(t × 2π − π/2) + 1) / 2
intensity = MIN + sine × (MAX − MIN)
```

`−π/2` の位相シフトにより t=0 で最小値から開始し、自然な「吸って吐く」リズムを表現する。

### ライフサイクル管理

- `setSelectedNode(nodeId)`: 既存グローを除去 → 新規 Mesh 生成 → シーンに追加
- `setSelectedNode(null)`: グローを除去、マテリアルを `dispose()`
- `rebuildScene()`: 冒頭で `setSelectedNode(null)` を呼び、古いオーバーレイを確実に除去
- `dispose()`: グロー Mesh とマテリアルの両方をクリーンアップ

## 代替案

### 1. インスタンス単位のカラー lerp

InstancedMesh の `setColorAt()` で選択カラムの色を時間変化させる方式。

**却下理由:** `setColorAt()` は `instanceColor` バッファを変更するため、毎フレーム `instanceColor.needsUpdate = true` が必要になる。また、カラー lerp だけではエミッシブグローを表現できず、視覚的に「洗い出された」印象になる。

### 2. カスタムシェーダー + per-instance attribute

インスタンスごとに `selected` フラグを attribute として持ち、フラグメントシェーダーでエミッシブ出力を分岐させる方式。

**却下理由:** 単一ノードの選択フィードバックに対して過度に複雑。シェーダーのメンテナンスコストが高く、`MeshStandardMaterial` のライティング機能（環境光、リムライト）を再実装する必要がある。将来的に複数選択が必要になった場合の拡張候補として記録しておく。

### 3. インスタンスの取り外し・再追加

選択カラムを InstancedMesh から除外し、独立した Mesh として再生成する方式。

**却下理由:** InstancedMesh のインスタンスバッファをコンパクション（詰め直し）する必要があり、実装の複雑さに対してメリットが見合わない。選択解除時に元のインスタンスを復元するロジックも煩雑になる。

## 影響・トレードオフ

### メリット

- 3D シーン内で選択状態が一目で分かり、オービット操作中もコンテキストを失わない
- severity カラーエンコーディングを損なわない（透過オーバーレイのため元色が透ける）
- 既存の InstancedMesh アーキテクチャに侵入的な変更を加えない
- `Canvas2DRenderer` は no-op スタブのため、2D フォールバックに影響しない

### デメリット・リスク

- 選択のたびに `THREE.Mesh` を生成・破棄する。頻繁なクリックでは GC 負荷が発生しうる（ただしジオメトリは共有参照のため軽量）
- 2D フォールバックモードでは選択の視覚フィードバックがない状態が継続する

### パフォーマンスへの影響

- 追加ドローコール: +1（選択中のみ）。InstancedMesh が数百カラムを 1 コールで描画している中での +1 は無視できるレベル
- `updateGlow()` は `Math.sin()` 1 回のみで、フレームあたりの計算コストは極小
- マテリアルの `emissiveIntensity` 変更は GPU 側の uniform 更新のみで、バッファ再アップロードは発生しない

## 今後の課題

- Canvas2D レンダラーでの選択ハイライト実装（リングやアウトラインで同等の効果を検討）
- 複数選択対応が必要になった場合、カスタムシェーダー方式（代替案 2）への移行を検討
- グローパラメータのユーザー設定可能化（アクセシビリティ観点：光過敏症への配慮として無効化オプション）
