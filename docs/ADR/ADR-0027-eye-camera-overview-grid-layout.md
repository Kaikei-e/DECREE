# ADR-0027: eye 初期カメラビューの俯瞰化とクラスタ内グリッドレイアウトの導入

## ステータス

Accepted

## 日付

2026-03-16

## コンテキスト

ADR-0026 で Threat Skyline（カラムノード＋ディストリクトプレート）に移行した直後、以下の 2 つの視認性課題が発覚した:

- **カメラ初期位置がカラム群の内部に埋もれる**: 従来の `resetView()` は球体ノードの Y 座標範囲に基づいてカメラ距離を計算していた。カラム化後はノードの `position.y` がスコア値のままだが、実際の描画高さは `getColumnHeight()` で拡張されるため、計算上のバウンディングボックスよりカラムが高くなり、初回表示時にカメラがカラム群に突っ込んだ状態になっていた
- **クラスタ内のノードが Z 軸方向に広がらない**: 従来のレイアウトは `jitterX`（範囲 `2.0`）と `jitterZ`（範囲 `1.0`）でノードを散らしていたが、Z 方向の散布が不十分でノードが X 軸上の一列に並ぶ傾向があった。カラム化により「列の奥行き」が視覚的に重要になったが、Z 方向に広がりがないためディストリクトプレート上がスカスカに見え、スカイラインの立体感が損なわれていた

これらの課題は ADR-0026 のカラム＋ディストリクト設計を活かすための必須の補正であり、レイアウト計算とカメラ配置の両方を見直す必要があった。

## 決定事項

1. クラスタ内のノード配置をランダムジッター主体から**グリッドレイアウト**に変更し、X・Z 両軸に均等に展開する
2. カメラの初期ビューを**俯瞰アングル**に変更し、スカイライン全体を見下ろせる位置に配置する
3. シーンバウンディングの計算基準を Y 座標範囲から**カラム実描画高さ（`maxHeight`）と XZ 平面スパン**に変更する

## 実装の概要

### layout.ts — グリッドレイアウトの導入

**ジッター範囲の縮小:**

```typescript
// Before
export const JITTER_RANGE = 2;

// After
export const JITTER_RANGE = 0.35;
```

ジッターは重畳防止の微小なゆらぎとしてのみ機能するよう大幅に縮小した。

**グリッド配置アルゴリズム:**

```
CLUSTER_GRID_X_SPACING = 0.9
CLUSTER_GRID_Z_SPACING = 0.9
```

1. クラスタごとにフィンディングを集約し、DECREE Score 降順 → `instance_id` 昇順でソート
2. `columnCount = max(2, ceil(sqrt(n)))` で正方形に近いグリッド列数を決定
3. 各ノードに `col`（列）と `row`（行）を割り当て
4. 最終座標: `x = cluster.centerX + (col - (columnCount-1)/2) × 0.9 + jitter`、`z = (row - (rowCount-1)/2) × 0.9 + depDepth × Z_SCALE + jitter`

ソート済みのためスコアの高いノードが左上に集まり、ディストリクト内を左上→右下の順で読むとスコア降順になる。

### ThreeSceneRenderer.ts — getColumnHeight() の抽出

カラム高さの計算をインライン式から専用メソッドに抽出:

```typescript
private getColumnHeight(node: GraphNode): number {
    return Math.max(MIN_COLUMN_HEIGHT, 1 + node.decreeScore * 1.1);
}
```

`rebuildScene()` と `getSceneBounds()` の両方で同一ロジックを使用し、描画高さとバウンディング計算の乖離を根本的に解消した。

### ThreeSceneRenderer.ts — getSceneBounds() の再設計

**Before（Y 座標ベース）:**

```typescript
{ cx, cy, spanX, spanY }  // Y = ノードの position.y 範囲
```

**After（XZ 平面 + カラム実高さベース）:**

```typescript
{ cx, cz, maxHeight, spanX, spanZ }  // maxHeight = 最も高いカラムの描画高さ
```

Y 軸のスパンではなく、XZ 平面のスパンとカラムの最大描画高さを個別に管理する。これによりカメラ距離・仰角の計算がカラムの実際の見た目に一致する。

### ThreeSceneRenderer.ts — resetView() の俯瞰化

**カメラ配置の変更:**

| パラメータ | Before | After |
|---|---|---|
| 距離 | `max(distY, distX, 15)` — Y/X スパンからの FOV 逆算 | `max(horizontalSpan × 1.45, maxHeight × 1.35, 18)` |
| 高さ（Y） | `cy + spanY × 0.1`（ほぼ正面） | `max(CAMERA_VERTICAL_PADDING=9, maxHeight × 0.7)` |
| 水平オフセット | なし | `cx + horizontalSpan × 0.32`（右斜め） |
| 注視点 | `(cx, cy, 0)` | `(cx, maxHeight × 0.35, cz)` |

カメラを右斜め上方に配置し、スカイラインを俯瞰するアングルにした。注視点はカラム群の高さの 35% に設定し、最も高いカラムの頂点と床面の中間付近を見る構成にした。

**`setGraphModel()` でのリセットビュー自動呼び出し:**

グラフモデル設定時に `this.resetView()` を呼び出すようにし、初回データ投入時に適切な俯瞰カメラが自動適用されるようにした。

**`setViewPreset()` の修正:**

`bounds.cy`（旧 Y 中心）を `bounds.maxHeight * 0.35` に置換し、`bounds.spanY` を `bounds.spanZ` に置換。Top / Front プリセットもカラム描画高さベースの計算に統一した。

### テスト

**layout.test.ts:**

- **グリッド展開テスト**: 12 ノードの単一クラスタに対し、`xValues.size > 2` かつ `zValues.size > 2` を検証。X・Z 両軸に十分な広がりがあることを保証

**ThreeSceneRenderer.test.ts:**

- **俯瞰カメラテスト**: `setGraphModel()` 後に `animateCamera` の最終呼び出し引数を検査し、`position.y > 8`（十分な高度）かつ `position.z > 10`（十分な後退距離）を検証

### データモデルへの影響

なし。フロントエンド（eye）のみの変更であり、fact / resource / projection いずれの層にも影響しない。

### 可視化への影響

Three.js WebGLRenderer のカメラ配置とノードレイアウト計算に変更がある。Sigma.js フォールバックは 2D レンダリングのため影響を受けない。`computeLayout()` は共通レイアウトエンジンだが、Z 軸方向のグリッド展開は 3D 表示でのみ視覚的な効果を持つ。

## 代替案

### A: カメラ距離のみを拡大し、レイアウトは変更しない

カメラを遠ざけるだけで「カラムに埋もれる」問題は解決するが、ノードが X 軸上の一列に並ぶ問題は残る。ディストリクトプレートの奥行きが活かされず、スカイラインの立体感が不足するため不採用。

### B: 物理シミュレーション（force-directed）でノードを自動配置する

`d3-force` 等の力学モデルでノード間の斥力を計算し、自然な分布を得る案。計算コストが高く、ノード数が増えた場合のフレーム遅延リスクがある。また配置が非決定的になるため、同じデータでも再描画ごとにレイアウトが変わり、ユーザーの空間記憶を阻害する。グリッドレイアウトは決定的で軽量なため採用した。

### C: Z 座標にジッター範囲を拡大するだけで対応する

`JITTER_RANGE` を Z 方向に大きくする案。ランダム配置のためノード間隔が不均一になり、局所的な密集や空白が生じる。グリッドベースの均等配置の方がディストリクト内の空間効率と可読性が高い。

## 影響・トレードオフ

### メリット

- 初回表示でスカイライン全体を俯瞰できるため、「最も危険な地区はどこか」を即座に把握できる
- グリッドレイアウトによりディストリクトプレート上にカラムが均等に展開され、プレートの奥行きが視覚的に活用される
- DECREE Score 順のソートにより、グリッド内の左上→右下が自然なリスク降順の読み順になる
- `getColumnHeight()` の抽出により、描画高さとバウンディング計算のロジック重複が解消された
- カメラ計算がカラム実描画高さベースになり、ノード数やスコア分布が変わっても適切なフレーミングが維持される

### デメリット・リスク

- グリッドの格子状配置はランダムジッターによる「有機的な」見た目を犠牲にしている。ただし `JITTER_RANGE = 0.35` の微小ゆらぎにより、完全な格子感は緩和されている
- `setGraphModel()` で毎回 `resetView()` が呼ばれるため、ユーザーがカメラを手動調整した後にデータ更新が来ると位置がリセットされる。リアルタイム更新時の UX として要検討

### パフォーマンスへの影響

- グリッドインデックスの事前計算（`findingsByCluster` のソートとインデックス割り当て）は O(n log n) だが、フィンディング数は通常数百〜数千程度のためレイアウト計算時間への影響は無視できる
- カメラ配置の計算量は変更前と同等（ノード走査 1 回）

## 今後の課題

- データ更新時のカメラリセットをオプション化する（初回のみ自動リセット、以降はユーザーの明示的操作で）
- グリッド内のソート基準を DECREE Score 以外（重大度、EPSS、パッケージ名等）に切り替えられる UI の追加
- ディストリクトごとのズームイン時に、グリッドレイアウトをより密に再計算して詳細表示する LOD 連動
- `CLUSTER_GRID_X_SPACING` / `CLUSTER_GRID_Z_SPACING` をノード数に応じて動的に調整し、大規模クラスタでの視認性を最適化する
